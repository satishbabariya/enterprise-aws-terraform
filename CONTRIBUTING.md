# Contributing

Thanks for considering a contribution. This repo is a Terraform template for
enterprise AWS organizations, so contributions are evaluated against three
things in order:

1. **Does it preserve compliance?** Every change must keep CIS, SOC2, PCI-DSS,
   and HIPAA controls intact (see `docs/compliance-matrix.md`).
2. **Does it preserve safety?** No state changes that could destroy production
   data. No removal of detective controls. No backward-incompatible module
   changes without an upgrade path.
3. **Does it follow existing patterns?** New modules look like existing
   modules. Account roots look like existing account roots.

## Local Setup

```bash
brew install terraform tflint pre-commit terraform-docs checkov
pre-commit install
```

## Workflow

1. Fork and clone, or create a feature branch.
2. Make your changes, following the patterns in existing modules.
3. Run `terraform fmt -recursive` and `pre-commit run --all-files`.
4. For module changes: ensure `terraform validate` passes in the module dir.
5. For account changes: ensure `terraform validate` passes in the account root.
6. Open a PR using the template - fill in every section.
7. CODEOWNERS will be auto-assigned for review.

## Module Conventions

Each module under `modules/<name>/` MUST have:

- `versions.tf` - Terraform + provider version constraints
- `variables.tf` - All inputs with `type`, `description`, defaults, and validation where applicable
- `main.tf` - Resources only
- `outputs.tf` - Anything another account might consume
- `README.md` - Auto-generated by terraform-docs (optional but recommended)

Naming:
- Resource names use `${var.org_name}-${var.account_name}-...`
- IAM role names always include the org prefix
- KMS keys always have an alias

Defaults:
- Encryption is always required, never optional
- Public access is always blocked at the resource level
- Logging is always enabled
- Tag inputs accept additional tags, never replace required ones

## What NOT to Submit

- New modules duplicating existing functionality (e.g., another VPC module)
- Modules that make security weaker by default (encryption optional, public access toggleable)
- Application-level resources (RDS instances, EKS clusters configured for a specific app) - those belong in workload-specific repos
- Backward-incompatible changes to existing module interfaces without a deprecation notice in `CHANGELOG.md`
- Personally identifying information, credentials, or real account IDs in commits

## Reporting Security Issues

See `SECURITY.md` - do not open public issues for security problems.
